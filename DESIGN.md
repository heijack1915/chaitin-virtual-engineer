## 一、产品概述

### 1.1 产品定位

**Chaitin-Virtual-Engineer**（长亭虚拟工程师）是一款面向长亭科技产品运维场景的**通用工具平台**。

它的核心理念是：**程序本身是通用的，通过加载不同产品的知识库来获得该产品的操作能力**。

### 1.2 目标用户

- 长亭科技技术支持工程师
- 渠道合作伙伴
- 终端客户运维人员

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| **降本增效** | 自动化安装、卸载、升级、排错，减少人工干预 |
| **知识传递** | 将专家经验封装为知识库，可复制、可传递 |
| **即插即用** | 加载知识库即可操作对应产品，无需复杂配置 |
| **离线可用** | 无网络环境下仍可基于本地知识库工作 |
| **AI增强** | 复杂场景下接入大模型，智能分析并给出解决方案 |

## 三、功能模块详解

### 3.1 主机连接管理

**功能描述**: 管理多个目标主机，支持SSH连接、凭据存储、连接测试。

**数据结构**:
```go
type Host struct {
    ID       string `json:"id"`
    Name     string `json:"name"`
    IP       string `json:"ip"`
    Port     int    `json:"port"`      // 默认22
    Username string `json:"username"`
    Password string `json:"password"`  // 加密存储
    Status   string `json:"status"`    // online/offline/error
}
```

**功能列表**:
- 添加/编辑/删除主机
- 测试连接
- 批量操作
- 凭据加密存储（脱敏显示）

### 3.2 操作面板

**功能描述**: 提供一键式操作，执行安装/卸载/升级/排错。

**操作类型**:

| 操作 | 自动化内容 |
|------|-----------|
| **安装** | 环境检测 → 传输安装包 → 执行安装 → 验证状态 |
| **卸载** | 备份配置 → 停止服务 → 清理目录 → 验证 |
| **升级** | 检查版本 → 传输升级包 → 执行升级 → 验证 |
| **排错** | 收集日志 → AI分析 → 定位问题 → 提供方案 |

**执行流程**:
```
用户选择操作 → 加载知识库 → 解析步骤 → 执行命令 → 实时输出 → 结果汇总
```

### 3.3 知识库管理（LLM Wiki 模式）

**设计理念**: 采用 Andrej Karpathy 提出的 LLM Wiki 模式，替代传统 RAG 向量检索。

**核心优势**:
- 预编译结构化 Wiki，后续推理基于 Wiki 而非原始文档
- 知识累积产生"复利效应"
- Token 消耗减少 95%
- 纯 Markdown 文件，离线完全可用，更易分发

**LLM Wiki 三层架构**:

```
┌─────────────────────────────────────────────────────┐
│  原始素材层 (Raw Sources)                            │
│  存放不可变的原始文档 - 模型仅读取不修改              │
│  格式: PDF / Markdown / HTML / TXT                   │
└─────────────────────────┬─────────────────────────────┘
                          │ LLM处理
                          ▼
┌─────────────────────────────────────────────────────┐
│  Wiki 表现层 (The Wiki)                              │
│  模型生成的 Markdown 文件目录                         │
│  - 实体页面 (entity/*.md)                            │
│  - 概念总结 (concepts/*.md)                          │
│  - 故障案例 (cases/*.md)                            │
│  - 操作手册 (guides/*.md)                            │
│  模型拥有完全写权限，持续更新维护                     │
└─────────────────────────┬─────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│  模式规范层 (The Schema)                            │
│  CLAUDE.md / AGENTS.md - 定义 Wiki 组织规范          │
└─────────────────────────────────────────────────────┘
```

**知识库压缩包结构**:
```
产品名_知识库_v1.0.zip
├── manifest.json           # 元信息
│   {
│     "name": "产品名",
│     "version": "1.0.0",
│     "created": "2026-04-24",
│     "author": "Matrix Agent",
│     "schema": "CLAUDE.md"
│   }
├── CLAUDE.md              # Wiki 组织规范（模式规范层）
├── sources/               # 原始素材层
│   ├── install.md
│   ├── troubleshoot.md
│   └── commands.md
└── wiki/                  # Wiki 表现层（LLM 编译产物）
    ├── index.md           # Wiki 索引
    ├── entities/          # 实体页面
    │   ├── 主机.md
    │   ├── 探针.md
    │   └── 容器.md
    ├── concepts/          # 概念总结
    │   ├── 安装流程.md
    │   └── 故障排查方法论.md
    └── cases/            # 故障案例
        └── 容器重启问题.md
```

**知识库加载流程**:
```
导入压缩包
    ↓
验证 manifest 和 CLAUDE.md
    ↓
加载 Wiki 表现层（LLM 编译产物）
    ↓
如需增量更新，LLM 阅读 sources/ 并更新 wiki/
    ↓
加载完成，可直接查询 Wiki
```

**离线工作原理**:
- 不依赖向量数据库
- 直接读取 Markdown 文件
- 模式规范层定义查询模式匹配
- 无 AI 时按模式规范层定义的规则匹配

### 3.4 AI 接口层

**设计理念**: 遵循 OpenAI 兼容格式，支持任意提供商的 API。AI 在 LLM Wiki 模式中扮演"编译器"角色。

**配置结构**:
```go
type AIConfig struct {
    Provider   string `json:"provider"`   // minimax/openai/siliconflow/oneapi/custom
    APIURL     string `json:"api_url"`     // API地址
    APIKey     string `json:"api_key"`     // 密钥
    Model      string `json:"model"`       // 模型名称
    Timeout    int    `json:"timeout"`     // 超时秒数
}

type AIProviders struct {
    MiniMax struct {
        APIURL string
        Models []string
    }
    OpenAI struct {
        APIURL string
        Models []string
    }
    SiliconFlow struct {
        APIURL string
        Models []string
    }
    OneAPI struct {
        APIURL string
        Models []string
    }
}
```

**AI 在 LLM Wiki 中的工作流程**:

```
┌─────────────────────────────────────────────────────────┐
│  AI 工作模式                                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 增量摄取 (Ingest)                                    │
│     用户添加新素材 → AI 阅读 → 更新 Wiki 相关页面         │
│     新素材可能触发 10-15 个相关页面的修改                 │
│                                                         │
│  2. 闭环查询 (Query)                                     │
│     用户提问 → AI 基于 Wiki 回答                         │
│     深度分析结论 → 重新存入 Wiki 成为新页面               │
│                                                         │
│  3. 定期巡检 (Lint)                                       │
│     AI 定期检查 Wiki 健康度                               │
│     检查页面间逻辑矛盾、过期内容、缺失链接                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**AI 调用流程**:
```
本地知识库检索 → 构建Prompt → 调用AI API → 解析响应 → 返回结果
```

**离线模式**:
- 无网络或未配置AI时，使用本地规则引擎分析
- 基于知识库中的 `rules.json` 进行匹配

### 3.5 任务调度

**功能描述**: 管理长时间运行的任务，支持后台执行、进度跟踪、日志输出。

**任务状态**:
```
pending → running → completed/failed
```

### 6.3 CLAUDE.md（模式规范层示例）

```markdown
# CLAUDE.md - 知识库规范

## 角色定义
你是一个长亭科技产品运维专家助手。

## Wiki 组织结构
- `wiki/entities/` - 产品实体定义（主机、容器、服务等）
- `wiki/concepts/` - 概念和流程总结
- `wiki/cases/` - 故障案例和解决方案

## 增量摄取规则
当用户添加新素材时：
1. 阅读 sources/ 中的新文档
2. 更新 wiki/ 中相关页面
3. 如发现新概念，创建新页面
4. 维护页面间交叉引用

## 查询规则
- 用户提问时，直接读取相关 Wiki 页面
- 不要每次都去读 sources/ 原始素材
- 深度分析结论应存入 Wiki

## 页面格式规范
每个 Wiki 页面应包含：
- 标题
- 简要描述
- 详细内容
- 相关页面链接
- 最后更新时间
```

### 6.4 Wiki 页面格式示例

```markdown
# 容器启动失败

## 概述
描述此故障的通用信息。

## 症状
- cloudwalker 容器状态为 Restarting
- 日志显示唯一键冲突错误

## 根因
数据库迁移脚本 bug：遍历 se_bash 表时未按架构去重。

## 排查步骤
1. `docker logs --tail 100 cloudwalker`
2. `docker exec -it db psql -U cloudwalker -d cloudwalker -c "\d se_bash"`

## 解决方案
更新 se_bash 表的 target 字段，区分重复记录。

## 相关页面
- [[安装流程]]
- [[数据库结构]]

## 更新时间
2026-04-24
```

## 七、AI接口规范

### 7.1 标准OpenAI格式

```go
// 请求
{
  "model": "gpt-4o",
  "messages": [
```
